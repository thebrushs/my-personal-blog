# 个人学习记录应用PRD（重构版）

## 一、产品重构核心调整

### 1.1 架构调整核心思路

原PRD为纯前端本地存储的网页应用，重构后调整为**微信小程序（前端展示）+ Web管理后台（数据管理）** 分离架构，同时补充服务端与数据库设计，核心调整如下：

|重构点|原设计|新设计|
|---|---|---|
|终端形态|纯网页端（本地存储）|微信小程序（用户端）+ Web后台（管理端）|
|数据存储|浏览器IndexedDB本地存储|服务端数据库存储 + 小程序/后台数据同步|
|开发优先级|前端功能先行|数据库表设计 → 服务端接口 → 后台管理 → 小程序前端|
|核心定位|本地个人工具|轻量级多端学习记录工具（先小程序验证，后扩展）|
### 1.2 产品定位（重构后）

一款以**微信小程序为核心展示端**、Web页面为管理后台的轻量化个人学习记录工具，解决学习记录分散、多端管理不便的问题，先通过小程序验证核心需求，后续可扩展多端同步、分享等能力。

### 1.3 目标用户（补充后台使用场景）

|角色|特征|核心需求|
|---|---|---|
|普通用户（小程序）|碎片化学习，高频记录/查看|快速记录、日历视图展示、轻量化操作|
|管理端用户（Web）|批量整理、数据复盘|批量编辑/导出、数据统计、记录管理|

### 3.2 表设计核心调整说明

1. 放弃本地Base64存储图片，改为服务端存储图片URL（降低小程序/后台存储压力）；

2. 增加用户表（关联微信OpenID，适配小程序登录）；

3. 所有表增加逻辑删除字段（避免数据物理删除，便于恢复）；

4. 核心索引优化：`study_record`表增加`user_id+date`联合索引（高频按日期查询）。

## 四、功能模块拆分（小程序+Web后台）

### 4.1 微信小程序端（用户侧）

#### 4.1.1 核心功能

|模块|功能描述|简化说明（适配小程序轻量化）|
|---|---|---|
|登录模块|微信授权登录|仅需获取OpenID、昵称、头像，无需手机号|
|日历视图|当月日历展示、日期切换、记录标记|卡片式布局，有记录的日期显示标题，点击进入详情|
|记录编辑页|添加文字/截图/代码/视频链接|截图自动上传至服务端，代码仅支持3种主流语言高亮|
|AI标题生成|上传截图/输入文字后生成标题|调用服务端AI接口（小程序端不做本地OCR）|

**AI标题生成接口设计：**
- **输入**：图片URL或文字内容（至少一项）
- **输出**：生成的标题（限制128字符内）
- **失败处理**：生成失败时返回默认标题"学习记录+日期"
- **超时设置**：接口超时时间5秒，超时后使用默认标题

|数据同步|自动同步记录到服务端|编辑后实时同步，无网时本地缓存，联网后自动同步|

#### 4.1.2 数据同步冲突处理策略

1. **乐观锁机制**：`study_record`表包含`version`字段，每次更新时检查版本号，版本不匹配时拒绝更新并提示"数据已更新，请刷新"

2. **冲突检测**：通过`update_time`时间戳比对，后提交者收到冲突提示，可选择覆盖或放弃

3. **离线同步策略**：
   - 本地记录操作日志（增/删/改操作及时间戳）
   - 联网后按时间顺序依次同步至服务端
   - 同步失败时保留本地操作日志，支持重试

4. **离线缓存策略**：
   - 缓存容量：最多缓存100条记录（约10MB）
   - 缓存清理：同步成功后自动清理本地缓存，保留最近7天记录
   - 缓存优先级：优先缓存文字内容，图片仅缓存URL

#### 4.1.3 小程序适配要点

1. 操作流程简化：减少长按/批量操作，优先适配触摸交互；

2. 存储优化：图片不上传至本地，直接上传服务端生成URL；

3. 离线能力：核心记录本地缓存，联网后自动同步至服务端。

## 五、技术选型调整（适配多端架构）

### 5.1 小程序端

- 基础框架：微信原生小程序（或uniapp跨端框架）；

### 5.2 Web管理后台

- 基础框架：Vue3 + Element Plus（后台管理适配性强）；

- 图表可视化：ECharts（学习数据统计）；

- 权限控制：Vue Router + Pinia（路由鉴权）；

- 数据操作：Axios（调用服务端接口）。

### 5.3 服务端

- 框架：**Node.js + Express + TypeScript**（推荐，前后端技术栈统一，开发效率高）
- 数据库：MySQL 8.0+（支持JSON字段）+ Redis（用户Session、热点数据缓存、接口限流）
- 存储：阿里云OSS/腾讯云COS（存储图片）
- AI能力：调用第三方OCR/标题生成接口（如百度AI、讯飞AI）

**选择Node.js的理由：**
1. 前后端技术栈统一（Vue3 + TypeScript）
2. 开发效率高，适合快速迭代
3. 异步IO适合处理图片上传等IO密集型任务
4. 部署简单，资源占用少

## 七、MVP版本边界（重构后）

1. 小程序端仅保留核心记录/查看功能，无批量操作、复杂统计；

2. Web后台仅支持单用户管理，无多角色权限；

3. AI标题生成调用第三方接口，暂不做本地模型；

4. 图片上传限制：
   - 支持格式：JPG、PNG、GIF（不支持WebP，小程序兼容性问题）
   - 单张大小：≤5MB
   - 批量上传：最多5张
   - 图片压缩：上传前小程序端自动压缩至宽度≤1080px

5. 暂不支持多端分享、云端协作功能。
